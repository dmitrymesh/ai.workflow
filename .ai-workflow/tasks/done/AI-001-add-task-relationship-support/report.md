# Execution Report: AI-001

## Summary

Added first-class task relationship support to the Portable AI Task Protocol:

- New metadata fields per task: `parent`, `children`, `blocks`, `blocked_by`, `related`. New tasks created via `create` get them as empty defaults.
- New CLI subcommands: `link`, `unlink`, `show`. They keep both sides of every relationship in sync automatically (parent ↔ children, blocks ↔ blocked_by, related ↔ related), so managers/executors do not need to hand-edit YAML.
- `validate` now also checks that every relationship reference points to an existing task id, that no task references itself, and that `parent`/`children` and `blocks`/`blocked_by` are reciprocal. Errors are emitted with explicit messages and a non-zero exit code.
- `board` gained `Parent` and `Blocked By` columns; `list` shows the same fields inline. Blocked tasks and parent links are visible at a glance.
- Manager skill instructs how to split broad requests and create reciprocal links; reviewer skill instructs to verify relationship consistency.
- Both READMEs document the relationship model with a parent/child split example and a blocking example.

No new dependencies; the script remains stdlib-only Python.

## Assumptions

- `related` is treated as bidirectional (linking A→B mirrors B→A) for symmetry with the other relationship kinds, but is not validated for reciprocity since the task spec only mandates reciprocity for parent/child and blocks/blocked_by. A hand-edited one-sided `related` link will therefore not be flagged by `validate`; the CLI never produces such state.
- A task can have at most one parent. `link <task> parent <new>` replaces an existing parent and removes the stale entry from the previous parent's `children` list.
- The minimal in-script YAML parser/dumper continues to be sufficient: the new fields are scalars, `null`, or simple lists, all of which it already handles. No PyYAML dependency was introduced (the task forbids non-stdlib deps).
- The `templates/task.md` contract format was not changed; relationships live in `metadata.yaml` and are managed by the manager via the CLI per the updated `manager.md` skill.

## Known risks

- Existing task metadata files written before this change do not contain the new fields. They are auto-normalized to empty lists on read, but until a `move`, `link`, or `unlink` rewrites them, the on-disk file still lacks the keys. This is benign — `validate` and all CLI ops normalize on load. Only AI-001's pre-existing metadata had the fields populated; no other tasks exist in this repo today.
- `dump_simple_yaml` quotes string values but writes list items unquoted. This was already the case before this change and is preserved (the parser strips quotes either way).
- The minimal YAML dumper does not preserve unknown keys in any structural way beyond key/value/list — if a future field needs nested mappings, the dumper will need extending. Out of scope here.
